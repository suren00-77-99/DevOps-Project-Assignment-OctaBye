# DevOps Project Assignment - 8Byte

## Project Overview

This project demonstrates the implementation of a complete DevOps workflow on AWS using Infrastructure as Code, CI/CD automation, containerization, and monitoring.
The infrastructure is provisioned using Terraform, application deployments are automated through Jenkins, and the application is deployed as Docker containers on Amazon ECS Fargate.
The objective of the project is to automate infrastructure provisioning, application deployment, and monitoring while following DevOps best practices.
---

# Solution Architecture
The solution consists of three major parts:

### Infrastructure Provisioning
Terraform is used to create all AWS resources required for the application.

### Application Deployment
Jenkins automates the build, scan, and deployment process of the application.

### Monitoring and Logging
CloudWatch is used to monitor infrastructure and collect application logs.
---
# Infrastructure Provisioning
The entire AWS infrastructure was provisioned using Terraform modules.

## Virtual Private Cloud (VPC)
The first resource created was the VPC.
A VPC provides a dedicated network for all AWS resources. It allows us to isolate resources from other AWS customers and control network communication.
The VPC contains:
* Public Subnets
* Private Subnets
* Internet Gateway
* Route Tables
Public subnets are used for internet-facing resources such as the Application Load Balancer.
Private subnets are used for backend resources such as ECS tasks and the database.
This design improves security because the application containers and database are not directly exposed to the internet.

---
## Security Groups
Security Groups act as virtual firewalls.
They control which resources can communicate with each other.
Three security groups were created:

### ALB Security Group
Allows incoming HTTP traffic from users.

### ECS Security Group
Allows traffic only from the Application Load Balancer.

### RDS Security Group
Allows database traffic only from ECS containers.
This ensures that only trusted resources can communicate within the environment.
---
## Application Load Balancer
The Application Load Balancer is the entry point for users.
When a user accesses the application, the request first reaches the ALB.
The ALB then forwards the request to healthy ECS tasks.
The load balancer also performs health checks to verify that containers are running correctly.
Benefits:
* High Availability
* Traffic Distribution
* Health Monitoring
---

## Amazon Elastic Container Registry (ECR)
ECR is used to store Docker images.
After Jenkins builds the application image, the image is pushed to ECR.
ECS then pulls the latest image from ECR during deployment.
Using ECR provides:
* Secure image storage
* AWS integration
* Centralized image management
---
## Amazon ECS Fargate
Amazon ECS Fargate is used to run the application containers.
Instead of managing EC2 servers, Fargate provides a serverless container platform.
The ECS setup consists of:

### ECS Cluster
A logical grouping of ECS services.

### Task Definition
Defines:
* Docker image
* CPU allocation
* Memory allocation
* Container port
* Logging configuration
  
### ECS Service
Maintains the desired number of running containers.
If a container fails, ECS automatically starts a replacement container.

Benefits of ECS Fargate:
* No server management
* Automatic scaling capability
* Reduced operational overhead
---
## Amazon RDS PostgreSQL
Amazon RDS is used as the backend database.
The database runs inside private subnets and is accessible only from ECS containers.
Using RDS eliminates the need to manage database servers manually.

Benefits:
* Automated backups
* Managed maintenance
* Improved reliability
* Secure networking
---
## CloudWatch Log Groups
CloudWatch Log Groups collect logs generated by ECS containers.
Whenever the application generates logs, they are automatically stored in CloudWatch.
This helps during troubleshooting and monitoring activities.
---
## Terraform Remote Backend
Terraform state management is configured using:

### Amazon S3
Stores the Terraform state file.
The state file tracks all resources managed by Terraform.

### DynamoDB
Provides state locking.
This prevents multiple users from modifying infrastructure simultaneously.

# Application Deployment
The application deployment process is fully automated using Jenkins.

---
## Source Code Checkout
Jenkins retrieves the latest application code from GitHub.
This ensures deployments always use the latest version of the application.

---
## Unit Testing
Before building the application image, Jenkins validates the Python application.
This helps identify syntax errors before deployment.

---
## Docker Image Build
The application is packaged into a Docker image.
Containerization ensures that the application behaves consistently across all environments.

Benefits:
* Portability
* Consistency
* Easy deployment

---

## Trivy Security Scan
Before deployment, the Docker image is scanned using Trivy.

This identifies:
* Critical vulnerabilities
* High-risk packages
* Security issues

This step improves the security of the deployment pipeline.

---

## Push Image to Amazon ECR

After the image passes validation, Jenkins pushes the image to Amazon ECR using the latest tag.
This image becomes available for deployment to ECS.

---

## Manual Approval
A manual approval stage is included before production deployment.
This prevents accidental deployments and provides deployment control.

---

## ECS Deployment
Jenkins triggers a new ECS deployment.

ECS then:
1. Pulls the latest image from ECR.
2. Starts new containers.
3. Performs health checks.
4. Replaces old containers.
5. Routes traffic through the ALB.

This ensures zero manual intervention during deployments.

# Monitoring and Logging
Monitoring is implemented using Amazon CloudWatch.
CloudWatch collects both metrics and logs from AWS resources.

---

## ECS Monitoring
The following ECS metrics are monitored:

* CPU Utilization
* Memory Utilization
* Running Tasks
These metrics help identify performance issues.
---

## Application Logs
Application logs generated by containers are automatically sent to CloudWatch Log Groups.
These logs are used for:
* Troubleshooting
* Error analysis
* Operational monitoring
---

## RDS Monitoring
CloudWatch is used to monitor database performance metrics such as:
* CPU Utilization
* Database Connections
* Storage Usage
---
# Security Implementation
Several security best practices were implemented throughout the project.
The application containers and database run inside private subnets.
Security Groups restrict communication between AWS resources.
Terraform state files are stored securely in S3.
DynamoDB prevents concurrent Terraform modifications.
Docker images are scanned using Trivy before deployment.
Manual approval stages are used before applying infrastructure and deploying applications.
